## Categories

| Secret | Where it lives | Rotation |
|---|---|---|
| `SUPABASE_SERVICE_ROLE_KEY` | Vercel env + 1Password | Quarterly |
| API key hashes (partner keys) | `api_keys` table | Rotated by partners |
| Agent token hashes | `agents` table | Rotated per partner discipline |
| Ed25519 private keys | Never (customer-held) | Per partner |
| Webhook secrets | `webhooks.secret_hash` | Rotated by partners |
| External provider secrets (Circle, Stripe, Persona) | Vercel env + 1Password | Quarterly or on compromise |
| Database connection strings | Vercel env (pulled by Supabase client) | Only on database migration |
| Signing keys (mandate JWTs, agent cards) | HSM | Yearly; rotation procedure below |

## Discipline rules

- **Never commit secrets.** Period. Pre-commit hooks scan for patterns. If you suspect you committed one:
  1. Rotate immediately
  2. Filter from git history with `git filter-repo`
  3. Force-push to clean up
  4. Notify #security

- **Never paste secrets in issue trackers, chat, or email.** 1Password share links instead.

- **Never request secrets to "debug."** If you can't reproduce without the secret, you're working on the wrong layer. Use test credentials.

## Adding a new secret

1. Generate the secret
2. Store in 1Password under the appropriate vault
3. Add to Vercel envs (dev, preview, production) — not the value in source
4. Add the key (not value) to `.env.example`
5. Document in this page if operationally significant

## Rotating

Secrets with automated rotation:
- Supabase service role key — reminder + dashboard rotation
- External provider keys — per-provider procedure in 1Password

Secrets with manual rotation:
- Mandate signing keys — quarterly playbook:
  1. Generate new keypair in HSM
  2. Enable "overlap mode" — both keys accepted
  3. Issue new mandates with new key
  4. Wait for old-mandate expiry
  5. Disable old key

## Incident response

If a secret may be compromised:
1. **Rotate immediately** — don't wait
2. Open incident channel
3. Audit access logs for the window of potential compromise
4. Determine blast radius (what did this key allow? who used it?)
5. Notify affected tenants / partners if relevant
6. Postmortem

## Key exfiltration detection

We monitor:
- Anomalous use patterns on API keys (new IP geography, new scope)
- Multiple simultaneous uses from different IPs
- Unusual call rates

Alerts fire to `#security-alerts`. Investigate promptly.

## Partner secrets

Partners hold their API keys, agent tokens, and webhook secrets. If they lose one:
- Rotate via `POST /v1/api-keys/:id/rotate` (if applicable)
- Revoke the lost one
- Issue new one — partner saves it once (shown one time)

We cannot recover a lost secret. This is a feature, not a bug.

## Annual audit

Security team runs annual:
- Secret inventory (what exists vs. what we expect)
- Rotation compliance (is every secret rotated per schedule?)
- Access review (who can read 1Password vaults?)
- Exfiltration detection drill

Gaps get tickets.
