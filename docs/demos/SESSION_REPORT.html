Round 11 — New Findings

F1. x402.org public facilitator has a concurrency bottleneck on its settle relayer

When 7 Sly agents submitted EIP-3009 settlements to https://x402.org/facilitator/settle in parallel, 4 of 7 failed with "replacement transaction underpriced". All 4 failed settlements used the same upstream relayer EOA (0xd407e409E34E0b9afb99EcCeb609bDbcD5e7f1bf), which submits the transferWithAuthorization call on behalf of the payer. The facilitator's relayer reuses a single nonce slot without proper pipeline management, so parallel settles collide. Running the same 4 settlements sequentially with a 3-second spacer succeeded 4-for-4. This is an upstream ecosystem limitation, not a Sly bug, but it has implications for any production agentic workload: agents issuing concurrent x402 payments through the public facilitator should serialize or back-off. Sly could solve this in two ways: (a) run its own facilitator with a nonce-aware relayer pool, or (b) add a client-side queue in the SDK that throttles concurrent settles.

F2. Sly + Pimlico bundler handled 10 concurrent smart-wallet UserOps with zero failures

Counterweight to F1. All 10 agents submitted UserOps to /smart-wallet/send-usdc essentially simultaneously — 3 warm wallets and 7 cold (first-deploy). Every single one landed on-chain. The Pimlico public bundler (public.pimlico.io/v2/84532/rpc) evidently has proper nonce management + mempool visibility across concurrent operations. This validates our bundler choice for Phase 2 and is exactly the kind of concurrency property we'd want in a production paymaster setup.

F3. First-deploy smart-wallet UserOps are ~13× slower than warm ones (63s vs 5s)

The median first-deploy UserOp took 63.6 seconds. The median warm UserOp took 4.7 seconds. The delta is the factory-deployment leg: the bundler has to validate the deployment, execute it, and then execute the transfer, all atomically. Not a bug — expected behavior — but important for UX framing: agent developers should expect a ~1-minute cold-start when an agent makes its very first on-chain action via smart wallet. After that, sub-5-second UserOps are the steady state.

F4. Claude Code subagent shell auth anomaly (unresolved)

During Round 11 setup, the 3 agents (DataMiner, CodeSmith, ResearchBot) whose auth_token_hash values were already correct in the DB ran fine via Claude Code subagents. The 7 agents whose hashes had been repaired direct-to-DB during Round 11 setup failed with "Invalid agent token" from subagent shells on the write endpoints, even though the exact same tokens worked from the main session's shell and from direct HTTP calls. Root cause unclear — possibly related to an API-side token cache interacting with direct-DB hash updates in a subagent-specific way. Workaround: the 7 agents' tests were executed inline via a TypeScript runner from the main session, producing equivalent results. Followup: investigate whether a deploy-triggered cache flush would have avoided this, and whether subagent Bash shells have any environment difference that triggers the mismatch. Tracked for post-session investigation; not blocking any user-facing functionality.

x402 Step 3 — ERC-4337 Smart Wallets (the "agent never touches anything" unlock)

Step 2 gave every agent a raw secp256k1 EOA. Useful, but raw EOAs need ETH for gas and can't rotate keys. Step 3 upgrades each agent to a Coinbase Smart Wallet whose owner is the existing EOA, deployed via the public factory at 0x0BA5ED0c6AA8c49038F819E587E2633c4A9F428a (same address on Base mainnet and testnet). Smart wallet addresses are CREATE2-deterministic — we know them before they're deployed, and the first on-chain interaction deploys the contract atomically with whatever operation is being performed.

Phase 1 — Derivation + ERC-1271 signing

  • Endpoint: POST /v1/agents/:id/smart-wallet derives the CREATE2 counterfactual address and stores it alongside the agent's EOA key.
  • Endpoint: POST /v1/agents/:id/smart-wallet-sign signs arbitrary messages/typed data via the smart wallet. Produces an ERC-1271-compatible wrapped signature (owner sig + replay-protection data) that the wallet's isValidSignature() function accepts on-chain.
  • Verified: DataMiner's smart wallet signature accepted by publicClient.verifyMessage() with proper ERC-1271 + ERC-6492 (counterfactual) verification — before the contract was deployed.

Phase 2 — UserOperation execution via Pimlico bundler

  • Endpoint: POST /v1/agents/:id/smart-wallet/send-usdc — single API call that: decrypts the agent's owner key from credential-vault, constructs a UserOperation containing a USDC.transfer calldata, submits it to the Pimlico public bundler (public.pimlico.io/v2/84532/rpc, no API key), waits for inclusion, returns the userOpHash + on-chain txHash.
  • The smart wallet pays for its own gas from its ETH balance (Phase 2 uses self-funded gas; full paymaster-sponsored gas-in-USDC is a follow-up once Sly has a funded paymaster).
  • Agents never see the owner key, never sign transactions, never call the bundler, never hold ETH. The entire ERC-4337 plumbing is hidden behind a single Sly API.

Bugs Found and Fixed

1. Environment column missing in agent token auth

auth.ts:423 didn't SELECT the environment column, silently defaulting every agent to test. Caused downstream handlers to fail for agents whose wallets were in live. Fix: one-line diff adding environment to the SELECT. Commit c7fd0a0.

2. No idempotency on A2A message/send

Same JSON-RPC id sent twice created 2 separate tasks and charged the caller twice. Fix: new idempotency_key column + unique index scoped by (tenant_id, agent_id), pre-check in handler, race-safe fallback via unique constraint violation catch. Commit 3e5a3ae.

3. Skill routing hijacked by intent regex

Explicit skillId: "translation" was being ignored in favor of regex-based intent extraction from message text, causing translation requests to return wallet balance checks. Fix: short-circuit in the router — when skillId is present, forward directly to that skill and skip intent extraction entirely. Commit 2edfa45.

4. Rating score scale mismatch

Code expected 0-100 scale but agents were submitting 0-5 star ratings. Score of 4 was mapping to "unacceptable" because the threshold was 40/100. Every Round 9 agent flagged this. Fix: auto-detect scale — scores in (0, 5] are treated as stars and multiplied by 20. Commit e072ec3.

5. A2A route silently accepted invalid bearer tokens

POST /a2a/:agentId treated bad auth as "anonymous caller" instead of rejecting. Masked 7 cases of agent hash corruption during Rounds 3-7. Fix: strict rejection (401) when a bearer token is present but fails verification. Absent token still allows anonymous (x402-compatible). Commit a5a3016.

6. Service fees and AP2 mandates bypassed on-chain settlement

Both flows created transfers with status: completed directly, skipping the async worker that handles on-chain execution. Fix: route both through authorizeWalletTransfer() so they become authorized and get picked up by the worker. Commit 3356317.

7. On-chain settlement for autonomous agents

resolveSettlementMandate was prematurely setting transfers to completed based on a local isOnChainCapable check, preventing the async worker from ever picking them up. Fix: let the worker make the capability decision itself. Commit 9e7ce20.

8. Agent hash data corruption (7 of 10 agents)

During marketplace setup, 7 agents had auth_token_hash values that didn't match their tokens in /tmp/marketplace-agents.json. Went undetected because /a2a/:agentId had permissive auth (now fixed). Resolution: recomputed SHA-256 hashes for all 10 agents and updated the DB directly.
